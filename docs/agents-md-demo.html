Making a Legacy Codebase AI-Friendly

How we built, tested, and evolved sub-AGENTS.md files for uhc-portal — and what we learned about AI-assisted documentation along the way.

The Problem

uhc-portal
554+
Files in cluster details alone

Mixed JS/TS, Redux + React Query, 5 coexisting refresh paths, 11 tabs

The Gap

AI tools explore code well but can't infer:

- Migration direction (Redux is legacy)

- Dependency boundaries (wizards never import from day-2)

- Team decisions (don't add to refreshRelatedResources)

The Approach

Sub-AGENTS.md files using the Linux Foundation AAIF standard:

- Scoped to directories (like .gitignore)

- Predictable headings AI tools recognize

- Overview (humans) + Constraints (AI)

Honest Assessment

Where Sub-AGENTS.md Helps

Architectural judgment calls

P4: Sonnet improved +1.0 when docs explained the 5-path refresh model and which paths are intentionally different

Dependency direction

Run 1 P8: +1.25 when baseline mischaracterized wizard-to-day2 coupling direction

Human onboarding

First real domain documentation for cluster details and wizards. New engineers can understand the refresh architecture without reading 800 lines of JSX

Where It Doesn't

Code structure comprehension

Models score 4.88+ baseline by exploring code. They don't need docs to understand file organization

Over-guidance risk

Sonnet regressed on P1 and P5: docs pushed it toward documented patterns instead of exploring independently

Code generation (untested)

We tested comprehension, not whether AI writes better code with docs. That's the real use case and a gap in our evaluation

Methodology Lessons

Evaluation Bias

Our "correct answers" were AI-generated (by Opus 4.6 with full codebase access). The AI judge compared AI responses against AI-generated answers. This creates circular validation — a model can score 5.0 by arriving at the same conclusions through code analysis alone, making the docs look unnecessary even if they'd prevent mistakes during code generation.

Prompting Patterns That Worked

PatternExampleWhy It Worked
Challenge the AI's output"I don't think ClusterDetailsTop.jsx is the only component"AI gives shallow first drafts. Human domain knowledge catches gaps
Ask for evaluation design upfront"How will the prompts be graded? Will it compare both against each other?"Forces rigor before running experiments
Confront uncomfortable results"It doesn't appear sub AGENTS.md is worth doing?"Leads to honest analysis instead of confirmation bias
Pivot when data says to"I'm curious about human usage of these files"Found the real value (onboarding docs) after AI-uplift proved marginal
Separate concerns with git branches"eval-baseline vs eval-with-docs branches"Clean A/B testing without manual file juggling
Iterate on format, not just content"Shouldn't Constraints follow the documentation router pattern?"Structure matters as much as content for AI consumption

What We'd Do Differently

Evaluation

- Test code generation, not just comprehension

- Use human grading for at least a subset of responses

- Add defect-injection tests (does the model catch anti-patterns?)

- Track real-world proxy: corrections per chat session over a sprint

Documentation

- Start with constraints only (skip structural descriptions)

- Focus on what code can't tell you: migration direction, team decisions

- Keep Overview section for human onboarding value

- Plan for maintenance: constraints change with team decisions, not code

Bottom Line

Sub-AGENTS.md files provide marginal measurable AI uplift for top-tier models (+0.06 for Opus). Their real value is dual: (1) documenting constraints and team decisions that code alone can't convey, and (2) serving as the first real domain onboarding documentation for the most complex areas of the codebase. The process of building and testing them — including the honest confrontation with disappointing results — was itself a valuable exercise in understanding how AI tools interact with our code.
